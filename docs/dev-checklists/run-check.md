# Проверка ./run.sh после крупных изменений

Цель: быстрая проверка, что локальный dev-стек запускается корректно после значительных изменений.

Когда: перед созданием PR, после мержа крупных изменений, или если CI показывает проблемы с интеграцией.

Базовые шаги (локально):

1. Запуск/рестарт сервисов:
   - ./run.sh --stop    # опционально остановить старые
   - ./run.sh

2. Быстрая проверка сервисов (ожидаемые порты: фронтенд 3000, бэкенд 3555, Postgres 5432, Redis 6379, MinIO 9000):
   - curl -fsS http://127.0.0.1:3555/ || echo "Backend not responding"
   - curl -I http://127.0.0.1:3000/ || echo "Frontend not responding"
   - pg_isready -h 127.0.0.1 -p 5432 || echo "Postgres not ready"
   - redis-cli -h 127.0.0.1 ping || echo "Redis not responding"
   - curl -fsS http://127.0.0.1:9000/ || echo "MinIO not responding"

3. Если что-то не так — собрать логи:
   - docker-compose logs --tail 200
   - ./run.sh --logs  # если есть helper, либо проверить логи .wasp/ и контейнеров

4. Быстрая функциональная проверка:
   - npm test -- --testPathPattern="test/e2e"    # опционально, или локальные unit-тесты
   - открыть /dashboard и выполнить базовый сценарий

5. PR чеклист (включить в описание PR при крупных изменениях):
   - [ ] ./run.sh запускается
   - [ ] Бэкенд отвечает на / (порт 3555)
   - [ ] Фронтенд загружается (порт 3000)
   - [ ] Основные интеграции (DB/Redis/MinIO) доступны

Примечание: позже можно автоматизировать эти проверки в CI (smoke test) — этот документ описывает ручную процедуру для разработчиков.
